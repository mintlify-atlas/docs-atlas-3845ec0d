---
title: cog.yaml Reference
description: Complete reference for configuring Cog models with cog.yaml
---

`cog.yaml` defines how to build a Docker image and how to run predictions on your model inside that image.

It has three main sections: [`build`](#build), [`image`](#image), and [`predict`](#predict).

## Example Configuration

```yaml
build:
  python_version: "3.11"
  python_requirements: requirements.txt
  system_packages:
    - "ffmpeg"
    - "git"
predict: "predict.py:Predictor"
```

<Tip>
Run `cog init` to generate an annotated `cog.yaml` file that can be used as a starting point for setting up your model.
</Tip>

## Top-Level Properties

<ParamField path="build" type="object" required>
  Describes how to build the Docker image your model runs in. See [Build Configuration](#build) for all available options.
</ParamField>

<ParamField path="image" type="string">
  The name given to built Docker images. If you want to push to a registry, this should also include the registry name.
  
  ```yaml
  image: "r8.im/your-username/your-model"
  ```
  
  If you don't set this, a name will be generated from the directory name. When set, you can run `cog push` without specifying the model name.
  
  <Note>
  r8.im is Replicate's registry, but this can be any Docker registry.
  </Note>
</ParamField>

<ParamField path="predict" type="string">
  The pointer to the `Predictor` object in your code, which defines how predictions are run on your model.
  
  ```yaml
  predict: "predict.py:Predictor"
  ```
</ParamField>

<ParamField path="train" type="string">
  The pointer to the `Predictor` object in your code that defines how training runs on your model.
  
  ```yaml
  train: "train.py:Trainer"
  ```
</ParamField>

<ParamField path="concurrency" type="object">
  Describes the concurrency capabilities of the model. Added in cog 0.14.0.
  
  <Expandable title="properties">
    <ParamField path="concurrency.max" type="integer" required>
      The maximum number of concurrent predictions the model can process. If this is set, the model must specify an async `predict()` method.
      
      ```yaml
      concurrency:
        max: 10
      ```
    </ParamField>
    
    <ParamField path="concurrency.default_target" type="integer">
      The default target for number of concurrent predictions. This setting can be used by an autoscaler to determine when to scale a deployment of a model up or down.
    </ParamField>
  </Expandable>
</ParamField>

<ParamField path="environment" type="array">
  A list of environment variables to make available during builds and at runtime, in the format `NAME=value`.
  
  ```yaml
  environment:
    - "DEBUG=true"
    - "MAX_WORKERS=4"
  ```
</ParamField>

<ParamField path="weights" type="array">
  A list of weight files or directories to include in the model.
  
  <Expandable title="properties">
    <ParamField path="weights[].name" type="string">
      A unique identifier for this weight entry.
    </ParamField>
    
    <ParamField path="weights[].source" type="string" required>
      Path to a weight file or directory (relative to cog.yaml).
    </ParamField>
    
    <ParamField path="weights[].target" type="string">
      Target path in the container (must be under /cache/). Defaults to /cache/\<basename\>.
    </ParamField>
  </Expandable>
  
  ```yaml
  weights:
    - source: weights/model.safetensors
      target: /cache/model.safetensors
  ```
</ParamField>

## Build Configuration

The `build` section describes how to build the Docker image your model runs in.

<ParamField path="build.python_version" type="string" required>
  The minor (`3.11`) or patch (`3.11.1`) version of Python to use.
  
  ```yaml
  build:
    python_version: "3.11.1"
  ```
  
  Cog supports Python 3.10, 3.11, 3.12, and 3.13. If you don't define a version, Cog will use the latest version of Python 3.13 or a version of Python that is compatible with the versions of PyTorch or TensorFlow you specify.
  
  <Note>
  These are the versions supported **in the Docker container**, not your host machine. You can run any version(s) of Python you wish on your host machine.
  </Note>
</ParamField>

<ParamField path="build.python_requirements" type="string">
  A pip requirements file specifying the Python packages to install.
  
  ```yaml
  build:
    python_requirements: requirements.txt
  ```
  
  This follows the standard [requirements.txt](https://pip.pypa.io/en/stable/reference/requirements-file-format/) format.
  
  <Note>
  Your `cog.yaml` file can set either `python_packages` or `python_requirements`, but not both. Use `python_requirements` when you need to configure options like `--extra-index-url` or `--trusted-host` to fetch Python package dependencies.
  </Note>
  
  ### Installing Git-hosted packages
  
  To install Git-hosted Python packages, add `git` to the `system_packages` list, then use the `git+https://` syntax:
  
  **cog.yaml:**
  ```yaml
  build:
    system_packages:
      - "git"
    python_requirements: requirements.txt
  ```
  
  **requirements.txt:**
  ```
  git+https://github.com/huggingface/transformers
  ```
  
  You can also pin Python package installations to a specific git commit:
  
  ```
  git+https://github.com/huggingface/transformers@2d1602a
  ```
  
  <Note>
  You can use a shortened prefix of the 40-character git commit SHA, but you must use at least six characters.
  </Note>
</ParamField>

<ParamField path="build.python_packages" type="array">
  <Warning>
  **DEPRECATED**: This will be removed in future versions, please use `python_requirements` instead.
  </Warning>
  
  A list of Python packages to install from the PyPI package index, in the format `package==version`.
  
  ```yaml
  build:
    python_packages:
      - pillow==8.3.1
      - tensorflow==2.5.0
  ```
  
  Your `cog.yaml` file can set either `python_packages` or `python_requirements`, but not both.
</ParamField>

<ParamField path="build.system_packages" type="array">
  A list of Ubuntu APT packages to install.
  
  ```yaml
  build:
    system_packages:
      - "ffmpeg"
      - "libavcodec-dev"
  ```
</ParamField>

<ParamField path="build.gpu" type="boolean" default={false}>
  Enable GPUs for this model. When enabled, the [nvidia-docker](https://github.com/NVIDIA/nvidia-docker) base image will be used, and Cog will automatically figure out what versions of CUDA and cuDNN to use based on the version of Python, PyTorch, and Tensorflow that you are using.
  
  ```yaml
  build:
    gpu: true
  ```
  
  <Note>
  When you use `cog run` or `cog predict`, Cog will automatically pass the `--gpus=all` flag to Docker. When you run a Docker image built with Cog, you'll need to pass this option to `docker run`.
  </Note>
</ParamField>

<ParamField path="build.cuda" type="string">
  Cog automatically picks the correct version of CUDA to install, but this lets you override it by specifying the minor (`11.8`) or patch (`11.8.0`) version of CUDA to use.
  
  ```yaml
  build:
    cuda: "11.8"
  ```
</ParamField>

<ParamField path="build.cudnn" type="string">
  Cog automatically picks the correct version of cuDNN to install, but this lets you override it for whatever reason.
  
  ```yaml
  build:
    cudnn: "8"
  ```
</ParamField>

<ParamField path="build.run" type="array">
  A list of setup commands to run in the environment after your system packages and Python packages have been installed. If you're familiar with Docker, it's like a `RUN` instruction in your `Dockerfile`.
  
  ```yaml
  build:
    run:
      - curl -L https://github.com/cowsay-org/cowsay/archive/refs/tags/v3.7.0.tar.gz | tar -xzf -
      - cd cowsay-3.7.0 && make install
  ```
  
  <Warning>
  Your code is _not_ available to commands in `run`. This is so we can build your image efficiently when running locally.
  </Warning>
  
  ### Using secret mounts
  
  Each command in `run` can be either a string or a dictionary with secret mounts:
  
  ```yaml
  build:
    run:
      - command: pip install
        mounts:
          - type: secret
            id: pip
            target: /etc/pip.conf
  ```
  
  You can use secret mounts to securely pass credentials to setup commands, without baking them into the image. For more information, see the [Dockerfile reference](https://docs.docker.com/engine/reference/builder/#run---mounttypesecret).
</ParamField>

<ParamField path="build.pre_install" type="array">
  A list of setup commands to run in the environment before your Python packages are installed.
  
  ```yaml
  build:
    pre_install:
      - apt-get update
      - apt-get install -y custom-package
  ```
</ParamField>

<ParamField path="build.sdk_version" type="string">
  Pin the version of the cog Python SDK installed in the container. Accepts a [PEP 440](https://peps.python.org/pep-0440/) version string. When omitted, the latest release is installed.
  
  ```yaml
  build:
    python_version: "3.12"
    sdk_version: "0.18.0"
  ```
  
  Pre-release versions are also supported:
  
  ```yaml
  build:
    sdk_version: "0.18.0a1"
  ```
  
  <Note>
  When a pre-release `sdk_version` is set, `--pre` is automatically passed to the pip install commands for both `cog` and `coglet`, so pip will resolve matching pre-release packages.
  </Note>
  
  <Warning>
  The minimum supported version is `0.16.0`. Specifying an older version will cause `cog build` to fail with an error.
  </Warning>
  
  The `COG_SDK_WHEEL` environment variable takes precedence over `sdk_version`. See [Environment variables](/config/environment-variables) for details.
</ParamField>

## Tests Configuration

<ParamField path="tests" type="array">
  A list of test cog commands to run.
  
  ```yaml
  tests:
    - predict:
        input:
          text: "hello world"
  ```
</ParamField>
